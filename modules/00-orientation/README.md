# Module 00 — Why Systems Thinking

**Phase 0 · Orientation**

Before any tools or diagrams: why does an engineer need systems thinking at all? This
module makes the case, sets the vocabulary, and establishes the working rhythm you'll use
for every module after it.

## Objectives

By the end of this module you should be able to:

1. Explain the difference between **reductionist** ("take it apart") and **systemic**
   ("look at the relationships") thinking, and say when each is appropriate.
2. Give at least three concrete examples where linear, cause-and-effect thinking **fails**
   an engineer — and name what was missing.
3. Define the working vocabulary: *system, element, interconnection, purpose, feedback,
   emergence.*
4. State, in one sentence, what you personally want out of this curriculum.

## Key concepts

- **A system is more than its parts.** A system is a set of *elements* interconnected in a
  way that produces its own pattern of behavior over time. Take the parts apart and the
  behavior disappears — it lived in the *relationships*, not the pieces.
- **Reductionism vs. holism.** Reductionism (decompose, isolate, optimize each part) is the
  default engineering move and it works beautifully for *complicated* problems. It fails for
  *complex* ones — where parts interact, adapt, and feed back on each other.
- **Emergence.** System-level behavior (a traffic jam, a cache stampede, a death-march
  project) that no single component "contains." It emerges from interaction.
- **Why linear thinking fails:** feedback, delays, and nonlinearity. Causes loop back into
  effects; consequences arrive late; small changes can produce large (or no) response.
- **Where engineers meet this daily:** an incident that cascades, a "fix" that makes things
  worse next quarter, a metric that gets gamed, a team reorg that reshapes the codebase.
