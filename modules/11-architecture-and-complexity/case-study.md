# Module 11 — Case Study

> **Your work.** Apply this module's lens to a real codebase or architecture you actually know.
> The goal isn't to indict the code — it's to see the *structure* that produces its behavior:
> debt as a stock, coupling as interconnection, decay as a loop. Use the stock-and-flow and
> causal-loop tools from Modules 02–03.

## The system

_Name one real codebase, service, or architecture you know well. Describe in a sentence what it
does and how old it is._

## Debt as a stock

_Draw technical debt here as a stock with its inflow (shortcuts) and outflow (paydown). What is
each flow actually made of in this system? Is the stock rising, falling, or flat right now?_

## Coupling as interconnection

_Redraw a slice of the architecture as relationships, not files. What flows across each edge?
Which node has the highest interconnection density, and what happens when it changes?_

## The reinforcing loop of decay

_Write the R1 loop for this system: harder to change → more shortcuts → ... → harder to change.
Fill the links with specifics. How fast is it compounding?_

## The missing (or present) balancing loop

_Is there a brake in place — a refactor budget, a CI complexity gate, an ownership rule? If so,
is it strong enough and fast enough? If not, describe the one you'd add and where you'd wire it._

## What I noticed

_Two or three sentences: what did the stock-and-loop view surface that a file-by-file review
would have missed?_
