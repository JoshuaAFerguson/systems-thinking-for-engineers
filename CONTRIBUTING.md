# Contributing

This is a personal **learning-in-public** repository — a record of one engineer working
through a systems-thinking curriculum. It isn't a community course, so "contributing" here
is lighter than usual. That said, thoughtful input is genuinely welcome.

## What's helpful

- **Corrections.** If a concept is stated wrong, a citation is off, or a chapter reference
  doesn't match the edition you have, open an issue. Getting the fundamentals right matters.
- **Better references.** A clearer paper, a canonical example, or a good free tool for a
  module — suggestions appreciated.
- **Sharper exercises or self-check questions.** If an exercise is vague or an archetype
  example would land better, say so.
- **Your own perspective.** Spotted an archetype or leverage point in a system you know?
  Issues and discussion are open.

## What to leave alone

The **`notes.md`** and **`case-study.md`** files in each module are deliberately the
learner's own work — synthesis in my own words and analysis of systems I know. Please don't
open PRs that fill those in. Suggestions on the *prompts* are fine; the *answers* are the point.

## How to raise something

- Small correction or suggestion → **open an issue**.
- Fix to the assignment side (README / exercises / self-check / reading list) → an issue or a
  small PR both work.

## Scope

Stay focused on the systems-thinking material and its engineering applications. This repo is
intentionally narrow; it isn't trying to be a general complexity-science reference.
